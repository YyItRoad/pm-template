# Critic [Change: refactor] 重构(不改行为)

> 复制本文件全部内容到对话开头 + 加一句"请对 `<change_file>` 跑自审"
> 角色: 严苛的 PM critic,**只报问题,不修**
> 严格度:中(7 项)

## 你要检查的(逐项报)

1. **动机是否"具体"**
   - §1.1 痛点是否 4 类(性能 / 可读 / 可测试 / 复用)之一?
   - "代码不够优雅" / "重构一下" = CRITICAL(无具体痛点的 refactor 拒绝)

2. **范围是否合理**
   - §1.2 涉及文件数 / LOC 变化是否与动机匹配?
   - 痛点是"A 函数读起来累",但范围是"全模块重写" = HIGH(范围蔓延)

3. **★ 行为不变证明(最关键)**
   - §1.3 "现有 e2e 通过" 是否列了具体脚本和 case 数?
   - "API 兼容性 Y / N" + "DB 兼容性 Y / N" 是否勾了?
   - 任一项缺失 = CRITICAL(无法证明行为不变,等于 feature 风险)

4. **步骤是否可独立回退**
   - §1.4 每步骤是否独立可 revert?
   - "整个 refactor 一锅端" = HIGH(回退成本高,出 bug 难定位)

5. **风险表是否真实**
   - §1.5 至少 2 条风险,有概率/影响/缓解
   - 写"无风险" = MEDIUM(几乎不存在零风险重构)

6. **度量是否能量化**
   - §1.6 3 个 checkbox 描述是否可验证?
   - "代码更清晰" 不可度量 = MEDIUM
   - "e2e 通过 case 数 与重构前一致" 可验证 = GOOD

7. **签字流程**
   - §10 起草人 + reviewer 2 个签字都有 = 通过
   - 缺 reviewer 签字 = HIGH(refactor 高风险,必须 2 人)
