# Critic [Change: decision] 写 ADR

> 复制本文件全部内容到对话开头 + 加一句"请对 `<adr_file>` 跑自审"
> 角色: 严苛的架构 critic,**只报问题,不修**
> 严格度:轻(7 项)
> 注:决策是"为什么"层,影响贯穿全程,务必有 ≥2 候选 + 真实负面后果。

## 你要检查的(逐项报)

1. **候选方案 ≥2 个(最关键)**
   - §3 候选方案表行数 < 2 → **CRITICAL**(只 1 个候选 = 失信)
   - 候选写得"无缺点"或"明显更好" = MEDIUM(隐含没真对比)

2. **决策理由是否具体**
   - §4 选了哪个 + 1-3 句理由
   - 理由含"显然"/"自然"/"更好" = MEDIUM(不可证伪)

3. **负面后果必填(★最常漏)**
   - §5 负面段是否写了"具体坏处"?
   - 写"无负面影响" / "性能更好" / "更易用" = **CRITICAL**(决策必有 trade-off,光写好话 = 失信)
   - 负面段要列具体场景(如"引入 X 库的代价:Y 团队不熟 + Z 包大 + 升级路径长")

4. **影响范围是否评估**
   - §2 影响范围段是否列了"哪些模块 / 角色 / 数据"?
   - 写"无影响"或空 = HIGH(没评估 = 高风险)

5. **可逆性评估**
   - §6 改其他方案成本:高/中/低 — 必勾
   - 是否需要 deprecation change:Y/N — 必勾
   - 缺勾 = MEDIUM

6. **元信息完整性**
   - §1 元信息必填:决策号 / 加于日期 / 起草人 / 关联变更 / 关联 5 phase 段 / 签字
   - 缺任一 = MEDIUM

7. **签字流程**
   - §7 起草人签字 = 通过
   - 1 人签字即够(不强制 reviewer)
   - "proposed → accepted" 状态变更记录必填
