# Critic [Phase 3b] 详细接口设计

> 复制本文件全部内容 + 加"请对 `<artifact_file>` 跑自审"

## 你要检查的

1. **接口清单 100% 对齐 Phase 2**
   - 提取 03b 所有接口名,跟 Phase 2 §3 求差集
   - 03b 多出 = CRITICAL(范围蔓延)
   - 03b 少 = CRITICAL(漏设计)
   - 差集必须为 0

2. **入参最小性**
   - 每个入参字段在 Phase 1 AC 中能找到出处?
   - 多 1 个冗余字段 = HIGH(防止"以后可能用")

3. **4 列必填**
   - 鉴权依赖 / 错误码清单 / 幂等策略 / 限频策略
   - 任意 1 列空白 = MEDIUM

4. **错误码齐全**
   - 每个接口 401 / 403 / 404 / 409 / 422 全部覆盖?
   - 漏 401/403 = HIGH(鉴权无定义)
   - 漏 409 = MEDIUM(冲突场景无定义)

5. **幂等策略**
   - 写操作(POST/PATCH/DELETE)必须有幂等策略
   - 缺 = HIGH(重复请求会双倍落账)

6. **末尾自查**
   - 03b 必须有"不在 Phase 2 §3 的接口数 = 0"声明
   - 没写 = HIGH

7. **范围蔓延**
   - 接口路径名有没有引入 Phase 1 故事没出现的能力(检查 path keyword)
   - 例:`/admin/...` path 在 Phase 1 没出现 = CRITICAL

8. **必填槽位(机械 grep,新加)**
   - grep `<!-- ANCHOR: api-list -->` 必命中
   - 接口清单表的列必含:接口 ID / 方法 / 路径 / 鉴权 / 请求体 / 响应体 / 错误码(7 列,缺一 = HIGH)
   - 抽查 1 个接口:7 列是否都填了(防"挂 anchor 列留空")

## 输出格式

- markdown 报告
- **每条 CRITICAL 必须列出具体接口 path**,不能空泛
