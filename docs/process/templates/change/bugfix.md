# 模板 [Change] Bugfix — 修缺陷

> 用途:修复已发现的具体 bug(行为偏离规格或崩溃)。
> 严格度:**中**(6 项 critic / 6 项 DoD / 1 人签字)。
> 禁止:借 bugfix 加新能力——那用 feature;借 bugfix 重构——那用 refactor。

---

## §1.1 缺陷描述 [必填]

- **标题**(一句话):___
- **严重度**:S0(线上崩溃/数据丢失)/ S1(主流程不可用)/ S2(边界场景)/ S3(体验瑕疵)
- **影响范围**:受影响功能 / 角色 / 数据范围
- **复现步骤**:
  1. ___
  2. ___
  3. ___

## §1.2 根因分析 [必填]

> 写**根因**,不写"现象"。3 个 why 收敛到代码层。

- **why 1**(为什么出现这个 bug):___
- **why 2**(为什么测试没拦住):___
- **why 3**(为什么设计允许了这种情况):___
- **根因定位**:文件 + 行号 / 函数名

## §1.3 修复方案 [必填]

- **改动文件**:`path/to/file.py:line-line`
- **修复内容**(用 diff 风格描述,不是自然语言):
  ```
  - old code
  + new code
  ```
- **为什么这样改而不是另一种**:___

## §1.4 回归测试 [必填]

> 必加"为这个 bug 写的测试",不能只验证修复有效,得验证**同类问题**不会再发。

- **新测试**:`tests/test_xxx.py::test_bug_NNNN`
- **覆盖的同类型 bug**:(列同类历史 bug 编号,如有)
- **手工 smoke 步骤**:___

## §1.5 已知风险 [必填,无则写"无"]

- 是否影响其他功能?
- 是否需要数据修复(线上已发生的数据)?
- 是否需要回滚预案?

## §10 critic + 签字(1 人)

- [ ] 变更 critic 报告无 CRITICAL/HIGH
- [ ] DoD 6/6 ✓
- [ ] 起草人 review + 签字: _______ (YYYY-MM-DD)
